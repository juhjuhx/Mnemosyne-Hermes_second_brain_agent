# 2026-06-05 Hermes 生態調研紀錄

> Research notes on the Hermes Agent ecosystem — the agent framework + memory layer
> + skill spec + tool protocol that v4 builds on.

---

## TL;DR

v4 replaces v3's self-built FastAPI agent + custom memory layer with:

- **Agent runtime**: [`NousResearch/hermes-agent`](https://github.com/NousResearch/hermes-agent) (MIT, 162K★, 2026 mid)
- **Memory backend**: [`AxDSan/mnemosyne`](https://github.com/AxDSan/mnemosyne) (MIT, SQLite-only, BEAM-friendly)
- **Skill spec**: [agentskills.io](https://agentskills.io) (Anthropic + Hermes native)
- **Tool protocol**: [MCP](https://modelcontextprotocol.io) (Anthropic, OpenAI adopted, Hermes native)

The combined stack is **~10x less code** than v3's self-built equivalent and
gets us a **community-maintained upgrade path**.

## 1. Hermes Agent

### What it is

`NousResearch/hermes-agent` is a **multi-model agent runtime** that:

- Accepts any OpenAI-compatible model API (Ollama, llama.cpp, vLLM, etc.)
- Implements the **MCP** tool protocol out of the box
- Loads skills from `agentskills.io`-format directories
- Has built-in memory plugin (Mnemosyne adapter available)
- Supports **sub-agents** (parallel task delegation)
- Ships a CLI (`hermes-cli`) and a web UI (optional)
- Has a built-in **API server** on `127.0.0.1:8642` (OpenAI-compatible)

### Key files we'd interact with

```
hermes-agent/
├── hermes/
│   ├── agent.py            ← main agent loop
│   ├── skills/             ← skill discovery + loading
│   ├── mcp/                ← MCP client + server framework
│   ├── memory/             ← memory backend abstraction
│   │   └── mnemosyne.py    ← Mnemosyne adapter
│   └── api_server.py       ← the OpenAI-compatible server
├── bin/
│   └── hermes-cli          ← terminal client
└── skills/                 ← bundled community skills
```

### Why we picked it

| Criterion | v3 self-built | Hermes Agent |
|---|---|---|
| Lines of code to maintain | ~5,000 (FastAPI) | ~50 (config) |
| Tool protocol | Custom JSON | MCP (standard) |
| Skill format | Custom | agentskills.io (standard) |
| Memory | Custom SQLite | Mnemosyne (pluggable) |
| Sub-agents | Not supported | Built-in |
| Web UI | Custom React | Optional built-in |
| Community | 1 user | 162K★ |
| Upgrade path | "rewrite it yourself" | `git pull` |
| License | (ours) | MIT |

### What we give up

- **Total control** — Hermes's agent loop is opinionated. We can't tweak every step.
- **Specific prompt engineering** — must use Hermes's prompt templates.
- **Custom memory backends** — must write a Mnemosyne adapter (community has one).

These are acceptable tradeoffs for the time saved.

### Hermetic boundaries

Hermes exposes hooks at the following boundaries, which we use:

- `pre_tool_call(tool_name, args)` — log every tool invocation
- `post_tool_call(tool_name, args, result)` — log results
- `pre_message_send(messages)` — final guardrail before sending to user
- `on_skill_loaded(skill_name)` — register skill-side state
- `on_error(exc)` — feed to ralph-watchdog

## 2. Mnemosyne

### What it is

`AxDSan/mnemosyne` is a **SQLite-only long-term memory layer** for AI agents.

Features:

- **Three-tier memory**: working (RAM) → episodic (SQLite, last 30 days) → semantic (Qdrant, infinite)
- **Time-decay weighting** — recent memories are weighted higher
- **Entity tracking** — remembers people, places, topics
- **BEAM-compatible** (65.2% on BEAM benchmark, per author)
- **LongMem-compatible** (98.9% on LongMem, per author)
- **SQLite-only storage** — no separate DB server
- **Hermes-native adapter** exists

### Why SQLite-only

- No separate DB process to manage
- Qdrant + SQLite together is enough (vectors + structured)
- Mnemosyne's structured data (entities, time-decay) goes in SQLite
- Mnemosyne's high-dim embeddings (semantic memory) go to Qdrant
- Backups = `sqlite3 .backup` + Qdrant snapshot

### Memory write path

```
Hermes agent loop
  └─→ on_skill_loaded / post_tool_call
       └─→ Mnemosyne.record(event_type, payload, timestamp)
            ├─→ SQLite (working + episodic tables)
            └─→ Qdrant (semantic vectors, named_vec.text)
```

### Memory read path

```
User query
  └─→ Hermes agent
       └─→ Mnemosyne.retrieve(query, top_k=20, recency_weight=0.3)
            ├─→ SQLite LIKE / FTS5 (episodic)
            ├─→ Qdrant search (semantic)
            └─→ Re-rank with time-decay
                 └─→ Top-K context injected into prompt
```

### Backup

- SQLite: `sqlite3 mnemosyne.db ".backup '/backup/mnemosyne-YYYY-MM-DD.db'"`
- Qdrant: POST `/collections/<name>/snapshots` then download

### Open question (Q07)

Do we need a single full snapshot of Mnemosyne, or S3-style incremental replication?
For v4, a single nightly snapshot is enough. Phase 5 may add streaming replication
if we run the system across geographically separated Macs.

## 3. agentskills.io (skill spec)

### What it is

A standardized directory format for "skills" an agent can load.

```
my-skill/
├── SKILL.md              # frontmatter + instructions
├── manifest.json         # MCP server config
├── tests.py              # pytest
├── README.md             # human-facing docs
└── (optional assets)
```

### SKILL.md format

```markdown
---
name: qdrant-search
description: Search Qdrant vector store for similar documents. Use when the user asks "find me documents about X" or "what's in my archive about Y".
---

# qdrant-search skill

This skill exposes a `qdrant.search` tool that returns top-K documents similar
to a natural language query.

## When to use

- User asks a semantic search question
- User wants to recall something they previously indexed

## When NOT to use

- User wants exact-match filename lookup (use `filesystem-search` instead)
- User wants a SQL query (use `sqlite-query` instead)

## Examples

See `examples/` directory.
```

### Why this format

- **Frontmatter** is machine-readable (name + description for routing)
- **Body** is human-readable instructions for the LLM
- **Manifest** is machine-readable MCP config
- **Tests** ensure the skill works
- **README** is human-facing

### Where skills live

- `src/hermes_skills/` — our 3 example skills
- `~/.hermes/skills/` — user-private skills (gitignored)
- `<repo>/.hermes/skills/` — repo-shared skills (CI-lintable)

## 4. MCP (Model Context Protocol)

### What it is

A **JSON-RPC protocol** for agents to call tools / read resources.

- Anthropic introduced it in 2024
- OpenAI adopted it in 2025
- Hermes Agent is MCP-native
- Ollama, llama.cpp, Qdrant all have MCP servers

### MCP servers we'd run

| Server | Purpose | Resource |
|---|---|---|
| `mcp-filesystem` | Read/write files in `/Volumes/RAID/AISecondBrain/` | filesystem |
| `mcp-qdrant` | Vector search, point CRUD, collection management | qdrant |
| `mcp-tags` | Tag taxonomy, add/remove/list | sqlite (tags.db) |
| `mcp-shell` | Run whitelisted shell commands (ffmpeg, etc.) | shell |
| `mcp-whisper` | Transcribe audio (delegates to whisper.cpp) | whisper |
| `mcp-eval` | Run eval suite, return metrics | eval |

### MCP server config (manifest.json)

```json
{
  "name": "mcp-qdrant",
  "version": "1.0.0",
  "description": "Qdrant vector store access",
  "transport": "stdio",
  "command": "python",
  "args": ["-m", "src.mcp_servers.qdrant_server"],
  "env": {
    "QDRANT_URL": "http://127.0.0.1:6333"
  },
  "tools": [
    {
      "name": "search",
      "description": "Search for similar documents",
      "input_schema": {
        "type": "object",
        "properties": {
          "query": { "type": "string" },
          "top_k": { "type": "integer", "default": 10 },
          "collection": { "type": "string", "default": "second_brain" }
        }
      }
    }
  ]
}
```
