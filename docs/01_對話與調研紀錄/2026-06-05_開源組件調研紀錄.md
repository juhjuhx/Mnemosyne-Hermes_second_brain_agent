## 1. Qdrant (vector store)

### What we use it for

- Store embeddings of all media (text + image)
- Named vectors: `text_vec(768) + image_vec(512)` — **v3 invariant**
- Single-node mode (we have 1 server)

### Key features in 2026

- `create_snapshot` API — atomic, zero-downtime backup
- `mmap` storage — fine for our scale (<1M vectors)
- Named vectors — built-in, no workaround
- Quantization — Scalar Quantization 4x, Product Quantization 8x
- HNSW index — default; M=16, ef_construct=100 good for our scale
- Payload indexes — fast filter on tags, dates, file types

### Versions to pin

- Qdrant 1.16.x (Q2 2026 release)
- qdrant-client 1.16.x (Python)

### Why not alternatives

- **Milvus** — needs a separate DB (etcd, MinIO). Too heavy for 1-node.
- **Weaviate** — similar story.
- **Chroma** — embedded mode is good but slower for our scale.
- **pgvector** — could work if we also want a relational store, but adds
  a Postgres dependency. v3 has SQLite for structured data; sticking with that.

### Backup strategy

- Nightly: `POST /collections/<name>/snapshots` → download to RAID
- Verify: load snapshot into a test container, run a smoke query
- Retention: keep 7 daily + 4 weekly + 12 monthly

### Open question (Q08)

If we ever shard Qdrant (Phase 5+), we need a different backup story.
For now, single-node + nightly snapshot is the answer.

## 2. PySceneDetect (video slicing)

### What we use it for

- Split videos into ≤10s segments for embedding
- Detect scene boundaries (camera cuts, action changes)
- Pre-process for vision-language model captioning

### 2026 status

- v0.7 released 2026-05-03
- AdaptiveDetector: F1=91.6% (best)
- ThresholdDetector: F1=88.2% (fast, good for static scenes)
- ContentDetector: F1=85.4% (legacy, kept for fallback)

### Configuration (v3 preserved)

- Use AdaptiveDetector with default thresholds
- Cap segment length at 10s
- Skip segments < 1s
- Embed the middle frame of each segment

### Pipeline

```
video file
  └─→ ffmpeg → extract keyframes (1 fps)
  └─→ PySceneDetect.AdaptiveDetector
       └─→ list of (start, end) tuples
            └─→ for each tuple: ffmpeg → middle frame → CLIP embed
                 └─→ write to Qdrant with video_id + segment_id
```

### Why not alternatives

- **TransnetV2** — better for TV shows, overkill for personal videos
- **AutoShot** — good for YouTube-style, less general
- **Decord** — faster, but no scene detection built-in

## 3. Open WebUI (frontend)

### What we use it for

- Primary chat UI (browse from any device on Tailscale)
- Model selector (switch between Hermes Agent and direct Qwen3.6)
- Document upload (triggers indexing)
- Conversation history

### 2026 status

- v0.6.x (Q2 2026)
- Native OpenAI API connections (just add Hermes's `http://127.0.0.1:8642`)
- Multi-user mode (we use single-user)
- Web search via SearXNG (we use local-only, so disabled)
- Code interpreter (sandboxed Python, useful for eval harness)

### Hermes integration (5-minute setup)

1. Open WebUI admin → Settings → Connections
2. Add OpenAI API:
   - URL: `http://127.0.0.1:8642/v1`
   - Key: any string (Hermes doesn't validate)
3. Set model name to "Hermes Agent"
4. Refresh model list — done.

### Why not alternatives

- **LibreChat** — newer, less stable
- **Chatbot UI** — simpler, fewer features
- **TypeChat** — terminal only, we have that via `hermes-cli`
- **Anything with cloud sync** — violates local-first invariant

### Mobile access

- Open WebUI has a PWA mode — install to home screen
- Tailscale on phone → access via `http://<m1-tailscale-ip>:3000`
- 2-3s latency on home Wi-Fi, acceptable

## 4. Tailscale (private mesh)

### What we use it for

- Connect M1 ↔ Workstation ↔ Phone
- 127.0.0.1 binding on all services — no public exposure
- Encrypted transport (WireGuard)
- DNS magic names (e.g. `m1`, `station`)

### 2026 status

- Free for personal use (up to 100 devices)
- Headscale (OSS alternative) viable if we want zero cloud dep
- ACLs: tag-based, can lock down M1 ↔ Workstation communication

### Configuration

- 2 nodes: `m1` (100.x), `station` (100.y)
- Magic DNS: enabled
- ACL: `m1` and `station` can talk to each other on specific ports
  (11434 Ollama, 8642 Hermes, 6333 Qdrant, 8080 llama-server, etc.)
- Funnel: disabled (no public exposure)
- Exit node: disabled

### Why not alternatives

- **ZeroTier** — works, but Tailscale's UX is better
- **Nebula** — DIY, more work
- **WireGuard direct** — too manual
- **Cloudflare Tunnel** — violates local-only

### Open question (Q03)

If we want to go **fully cloud-free**, switch to Headscale (the OSS control
server). Adds ~2 hours of setup. For v4 we use Tailscale's hosted control.

## 5. Ollama (M1 LLM serving)

### What we use it for

- Run Gemma 4 E2B on M1 with Metal
- Serve an OpenAI-compatible API on `127.0.0.1:11434`
- Hermes Agent connects to it as one of its "models"

### 2026 status

- v0.5.x (Q2 2026)
- Metal support: stable
- Modelfile: similar to Dockerfile, can specify system prompt + parameters
- Library: 100+ models pre-quantized, including Gemma 4

### Why not alternatives on M1

- **llama.cpp directly** — more setup, no model management
- **LM Studio** — GUI-only, doesn't run headless
- **GPT4All** — similar to LM Studio
- **vLLM** — needs Linux + CUDA, not M1

### Modelfile for Hermes routing

```dockerfile
FROM unsloth/gemma-4-e2b-it-GGUF:Q4_K_XL

# Hermes system prompt
SYSTEM """You are the M1 edge agent in the Personal AI Second Brain.
You handle fast, local tasks. For deep reasoning, you suggest
routing to the workstation agent."""

# Default sampling
PARAMETER temperature 0.7
PARAMETER top_p 0.9
PARAMETER num_ctx 8192
```

## 6. llama.cpp (workstation LLM serving)

### What we use it for

- Run Qwen 3.6 27B-PRISM-DQ on A770 with Vulkan
- OpenAI-compatible API via `llama-server` on `127.0.0.1:8080`
- Self-speculative decoding via bundled MTP head

### 2026 status

- b4567+ (Q2 2026)
- Vulkan backend: stable, supports MTP speculative decoding
- PRISM-DQ format: supported via `--speculative-model-draft auto`
- SYCL backend: also stable, but we use Vulkan (per user hardware)

### Build flags

```bash
cmake -B build \
  -DGGML_VULKAN=ON \
  -DGGML_NATIVE=ON \
  -DGGML_OPENMP=ON \
  -DLLAMA_BUILD_SERVER=ON \
  -DLLAMA_CURL=OFF \
  -DCMAKE_BUILD_TYPE=Release

cmake --build build --config Release -j 16
```

### Server command

```bash
./build/bin/llama-server \
  --model /home/user/models/Ex0bit/Qwen3.6-27B-PRISM-PRO-DQ.Q4_K_M.gguf \
  --speculative-model-draft auto \
  --draft-max 8 \
  --gpu-layers 99 \
  --ctx-size 32768 \
  --port 8080 \
  --host 127.0.0.1 \
  --parallel 4
```

### Why not alternatives

- **vLLM** — excellent, but no Metal/Vulkan path; needs CUDA
- **TGI (HuggingFace)** — same, CUDA-only
- **MLX** — Apple Silicon only, doesn't help on Linux workstation
- **exllamav2** — fast on CUDA, no Vulkan
