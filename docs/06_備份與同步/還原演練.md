# 還原演練 (Restore Drill)

> **目標**: 確保「最壞情況」(整個工作站掛了)時,使用者能在 **4-8 小時內** 從備份把整個第二腦重建出來。這個演練每季跑一次,並把結果記錄在本檔底部。

---

## 0. 為什麼需要演練?

備份沒有演練過 = 沒有備份。  
(Backup that has never been tested is not a backup.)

常見的「備份陷阱」:
- 還原時才發現 rclone crypt 密碼忘了
- 還原時才發現 OneDrive 的目錄結構不對
- 還原時才發現 Mnemosyne 的 DB 跟 Qdrant 的 snapshot 來自不同時間點
- 還原時才發現 `git tag` 對應的 commit 在某次 force-push 後不見了

演練的目的就是**在事情發生前**,把這些雷踩過一次。

## 1. 演練情境

### 情境 A: 單一檔案遺失 (5 分鐘)

**情境**: 使用者不小心刪了 `photos/2024-07-15_dog-beach.jpg`

**演練步驟**:
1. 在 `inbox/` 或 `archive/` 用 `find` 找
2. 從 L2 (RAID) 拉回:
   ```bash
   rsync -avh /Volumes/RAID/AISecondBrain/archive/photos/2024-07-15_dog-beach.jpg \
              ~/ai-brain-station/archive/photos/
   ```
3. 觸發 indexer 重新索引(只對這一個檔)
4. 驗證 Qdrant 內有此 file_id 的 point

**通過標準**: < 5 分鐘,使用者完全感覺不到中斷

### 情境 B: Mnemosyne DB 損壞 (10 分鐘)

**情境**: Mnemosyne 在寫入時斷電,DB 損壞

**演練步驟**:
1. 確認 Mnemosyne 已停止
   ```bash
   launchctl unload ~/Library/LaunchAgents/ai.brain.hermes.plist
   ```
2. 從 L2 還原最近一次備份
   ```bash
   LATEST=$(ls -t /Volumes/RAID/AISecondBrain/_backups/mnemosyne/*.db | head -1)
   cp "$LATEST" ~/ai-brain-station/hermes_config/mnemosyne/mnemosyne.db
   ```
3. 啟動 Mnemosyne
4. 跑 `sqlite3 mnemosyne.db "PRAGMA integrity_check"` 確認 DB 健康
5. 比對備份後新寫入的對話(看 Mnemosyne log 找最近的寫入時間)

**通過標準**: < 10 分鐘,DB integrity_check 通過,最多遺失備份後的對話(預期 < 24 小時)

### 情境 C: Qdrant 損壞 (30 分鐘)

**情境**: Qdrant 的 storage 出現 corruption

**演練步驟**:
1. 確認 Qdrant 已停止
   ```bash
   launchctl unload ~/Library/LaunchAgents/ai.brain.qdrant.plist  # M1
   # or
   systemctl stop ai-brain-qdrant-mirror  # workstation
   ```
2. 從 L2 還原最近一次 snapshot
   ```bash
   LATEST=$(ls -t /Volumes/RAID/AISecondBrain/_backups/qdrant/*.snapshot | head -1)
   curl -X POST "http://127.0.0.1:6333/collections/second_brain/snapshots/upload?priority=snapshot" \
        -H "Content-Type: multipart/form-data" \
        -F "snapshot=@${LATEST}"
   ```
3. 啟動 Qdrant,等待 recovery 完成
4. 跑 `curl http://127.0.0.1:6333/collections/second_brain` 確認 `vectors_count` 符合預期

**通過標準**: < 30 分鐘,vectors_count ≥ 備份時的 99%

### 情境 D: 工作站硬碟整個掛了 (4-8 小時)

**情境**: 工作站 SSD 故障,無法開機

**演練步驟**:
1. 購買新硬碟(若主機板也壞則買新主機)
2. 重灌作業系統(Linux 或 macOS)
3. 從 L3 (OneDrive) 拉回:
   ```bash
   # 安裝 rclone
   curl https://rclone.org/install.sh | sudo bash
   # 設定 crypt remote(從 1Password 取密碼)
   rclone config
   # 開始還原
   rclone copy onedrive-crypt:AI-Brain-Backup/archive /tmp/restore --progress
   ```
4. 從 L4 (git) 拉回程式碼:
   ```bash
   git clone https://github.com/juhjuhx/Mnemosyne-Hermes_second_brain_agent.git
   ```
5. 依 [`Phase_1_部署指南.md`](../02_方案文檔/Phase_1_部署指南.md) 重新部署
6. 從 Mnemosyne + Qdrant 備份還原
7. 用 `tag_release.sh` 打一個 `v4.0.X-{date}-disaster-recovery` tag 標記這次還原

**通過標準**: < 8 小時(包含購買硬碟的時間,若硬碟已備好則 < 4 小時)

### 情境 E: L2 (RAID) 損壞 (1-3 小時)

**情境**: RAID 控制器故障,2 顆硬碟中 1 顆有 bad sector

**演練步驟**:
1. 立即停止所有寫入到 L2 的 cron job
2. 評估 L2 損壞範圍(用 `smartctl -a /dev/sdX`)
3. 若可讀:直接從 L2 救資料
4. 若不可讀:從 L3 (OneDrive) 還原
5. 重建 RAID(更換壞硬碟,rebuild)

**通過標準**: < 3 小時,且資料遺失 < 7 天(Qdrant snapshot 保留 7 天)

### 情境 F: 整個雲端 + RAID 同時掛了 (最壞)

**情境**: 颱風淹水 + Microsoft 同時故障(機率極低)

**演練**: **不演練**。這種情境的 mitigation 是「分散風險到不同地理位置」,例如把 L3 改用 Google Drive 而非 OneDrive,或加一份 Backblaze B2。

## 2. 演練排程

| 情境 | 頻率 | 預計耗時 | 負責人 |
|---|---|---|---|
| A: 單檔 | 每月一次 | 5 分鐘 | 自動(shell script) |
| B: Mnemosyne | 每季一次 | 10 分鐘 | 自動 + 人工驗證 |
| C: Qdrant | 每季一次 | 30 分鐘 | 自動 + 人工驗證 |
| D: 工作站全毀 | 每年一次 | 4-8 小時 | 完整演練,可能需要離線半天 |
| E: RAID 損壞 | 每年一次(模擬) | 1-3 小時 | 完整演練 |
| F: 全毀 | **不演練** | n/a | 風險接受 |

## 3. 用 git tag 還原到特定版本

```bash
# 查看所有 tag
git tag -l "v4.*" --sort=-v:refname

# 還原到 v4.0.0-2026-06-12-phase1
git checkout v4.0.0-2026-06-12-phase1

# 重新安裝依賴
pip install -r tests/requirements.txt
pip install -r src/hermes_skills/*/requirements.txt

# 啟動對應版本的 services
# (注意:tag 之間的 service config 可能有 breaking change)
launchctl load ~/Library/LaunchAgents/ai.brain.hermes.plist
```

> ⚠️ 還原到舊 tag **不會** 自動還原 Mnemosyne 與 Qdrant 的備份——這些是獨立流程,見情境 B 與 C。

## 5. 自動演練腳本(給情境 A)

放在 `src/scripts/drill_file_restore.sh`:

```bash
#!/usr/bin/env bash
# drill_file_restore.sh — Monthly drill for scenario A (single-file restore)
# Runs the first Sunday of every month via cron.
#
# Verifies that a randomly-chosen file from L1 can be restored from L2.
# Fails (exits 1) if the restore does not match.

set -euo pipefail
L1="$HOME/ai-brain-station/archive"
L2="/Volumes/RAID/AISecondBrain/archive"
LOG="$HOME/ai-brain-station/logs/drill_$(date +%Y-%m).log"

ts() { date '+%Y-%m-%d %H:%M:%S'; }
log() { echo "[$(ts)] $*" | tee -a "$LOG"; }

# Pick a random small file
SAMPLE=$(find "$L1" -type f -size -1M | shuf -n 1)
log "Drill sample: $SAMPLE"

# Compute hashes
H1=$(shasum -a 256 "$SAMPLE" | cut -d' ' -f1)
H2=$(shasum -a 256 "$L2/${SAMPLE#$L1/}" | cut -d' ' -f1)

if [[ "$H1" != "$H2" ]]; then
  log "❌ DRILL FAILED: hash mismatch"
  log "  L1: $H1"
  log "  L2: $H2"
  exit 1
fi

log "✓ DRILL PASSED: $H1"
```
