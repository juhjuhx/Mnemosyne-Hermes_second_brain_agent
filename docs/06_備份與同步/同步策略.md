# 同步策略 (Sync Strategy)

> **核心原則**: 「記憶永不遺失」是 v3 的不變量。原始檔案永遠保留,所有同步都是 *additive*——絕不刪除、覆蓋前先備份、任何副本損壞都能從其他副本重建。

---

## 0. 摘要 (TL;DR)

| 層級 | 儲存 | 同步頻率 | 工具 | 保留 |
|---|---|---|---|---|
| L0: 主存 | 工作站 `~/ai-brain-station/inbox/` | 寫入即留 | rsync + ACL | 無限期 |
| L1: 工作目錄 | 工作站 `~/ai-brain-station/archive/` | 使用者手動 *tag* 後移動 | rsync | 無限期 |
| L2: 本機 RAID | `/Volumes/RAID/AISecondBrain/` | 每晚 02:00 | rsync | 無限期 |
| L3: 異地 (加密) | OneDrive `Personal Vault` | 每週日 03:00 | rclone crypt | 90 天 rolling |
| L4: 程式碼 | GitHub (本 repo) | git push | git | 永久 |

> **重要**: L1 → L2 → L3 全部 *append-only*。絕不允許 `rsync --delete` 跨層級,只能在層級 *內* 對舊快照做 rotation。

---

## 1. 3-2-1 規則的本地化實作

| 規則 | 實作 |
|---|---|
| **3** 份拷貝 | 主存 (L0) + RAID (L2) + OneDrive (L3) |
| **2** 種媒介 | SSD (L0/L1) + HDD RAID (L2) + 雲端 (L3) |
| **1** 份異地 | OneDrive Personal Vault(台灣境外節點) |

額外的「**第四份**」是 **GitHub 上的程式碼 repo**(只備程式碼、不備資料)。

## 2. 各層級的具體配置

### L0 + L1: 工作站本地

- **L0 路徑**: `~/ai-brain-station/inbox/` (使用者拖入的原始檔)
- **L1 路徑**: `~/ai-brain-station/archive/{text,image,audio,video,documents}/`
- **L0 → L1 觸發**: 使用者手動操作,或 indexer 標記 `suggested_move` 後由人工確認
- **v3 不變量**: **絕不自動搬移**——indexer 只會設定 `suggested_move` 欄位,從不自己改檔
- **可逆性**: 任何 L1 的檔案,只要 `file_id` 還在 SQLite 中,隨時可以還原到 L0

```bash
# 還原範例:把 file_id=abc123 從 archive 拉回 inbox
mv ~/ai-brain-station/archive/text/abc123.md ~/ai-brain-station/inbox/
```

### L2: 本機 RAID

- **路徑**: `/Volumes/RAID/AISecondBrain/{_backups,inbox,private,public,shared,media,_archive,index,logs}/`
- **容量**: 4TB RAID-1(可用 2TB)
- **同步工具**: `sync_raid.sh`(見本目錄)
- **頻率**: 每晚 02:00(cron)
- **保留**:
  - `_backups/qdrant/`: 7 個 nightly snapshot,到期自動刪除
  - `_backups/mnemosyne/`: 30 個 nightly 備份
  - `_backups/files/`: 完整 mirror(無 rotation)
  - `_archive/`: 年度歸檔(每年 12/31 凍結一份)

### L3: OneDrive 異地 (加密)

- **工具**: rclone + crypt remote(見 [`sync_onedrive.sh`](sync_onedrive.sh))
- **頻率**: 每週日 03:00(配合 cron)
- **保留**: 90 天 rolling,90 天前的歸檔移到 `archive-YYYY/` 子資料夾
- **加密**: rclone crypt,密碼存 1Password(`AI Brain Backup` 項目),不在腳本內硬編碼
- **上傳量**: 約 50-200GB/週(增量)

### L4: 程式碼

- **平台**: GitHub(`mnemosyne` repo)
- **觸發**: 每次 `git push` 後自動觸發 CI 跑 `structure-check.yml` 與 `markdown-link-check.yml`
- **歸檔**: 每個 phase 完成後打 `git tag v4.x.y-phaseN` 與 `_archive/2026-XX-XX-phaseN/` 雙重保存
- **資料夾規範**: 程式碼歸 `src/`,markdown 歸 `docs/`,資料庫備份歸 `_backups/`(在 `.gitignore` 中,不會進 git)

## 3. 各資料來源的備份方式

### 3.1 Mnemosyne SQLite

```bash
# Nightly 備份
python src/scripts/mnemosyne_backup.py \
    --db ~/ai-brain-station/hermes_config/mnemosyne/mnemosyne.db \
    --output /Volumes/RAID/AISecondBrain/_backups/mnemosyne/$(date +%Y-%m-%d).db
```

- 使用 SQLite 內建的 `.backup` API,**hot backup**(不需關閉 Mnemosyne)
- 30 天 retention,crontab 內含自動刪除
- 還原時直接 `cp .db mnemosyne.db` 即可

### 3.2 Qdrant 向量索引

```bash
# Nightly 快照
python src/scripts/qdrant_snapshot.py \
    --url http://127.0.0.1:6333 \
    --collection second_brain \
    --output /Volumes/RAID/AISecondBrain/_backups/qdrant/ \
    --keep-days 7
```

- 7 天 retention
- Qdrant 的 snapshot API 保證一致性(它會先暫停寫入再序列化)
- 還原時用 `qdrant restore` 指令

### 3.3 檔案 (PDF / 圖片 / 影片 / 音訊)

- **不在 v4 備份範圍內的**: 不需要(它們原本就在使用者自己的電腦與手機上)
- **在 v4 備份範圍內的**: `inbox/` 與 `archive/`(因為 indexer 標記後這些是「已經被第二腦消費過」的)
- 工具:`rsync -avh --no-perms --no-owner --no-group`(跳過 metadata 衝突)

### 3.4 程式碼 + 文件

- 主備份: Git
- 次備份: 每個 phase 結束時,把整個 repo 複製到 `_archive/2026-XX-XX-phaseN/`
- 這個 archive 是 *自包含的*(包含 LICENSE、README、所有 docs、所有 src)

## 4. Cron 配置範例

```cron
# /etc/cron.d/ai-brain
# 每天 02:00 同步到 RAID
0 2 * * *  user  /opt/ai-brain-station/src/scripts/sync_raid.sh >> /var/log/ai-brain/sync_raid.log 2>&1

# 每天 02:30 備份 Mnemosyne
30 2 * * *  user  /opt/ai-brain-station/src/scripts/mnemosyne_backup.sh >> /var/log/ai-brain/mnemo.log 2>&1

# 每天 03:00 備份 Qdrant
0 3 * * *  user  /opt/ai-brain-station/src/scripts/qdrant_snapshot.py ... >> /var/log/ai-brain/qdrant.log 2>&1

# 每週日 03:00 同步到 OneDrive
0 3 * * 0  user  /opt/ai-brain-station/src/scripts/sync_onedrive.sh >> /var/log/ai-brain/sync_od.log 2>&1
```

> 備份順序刻意錯開(02:00 → 02:30 → 03:00 → 03:00週日),避免同時 I/O 風暴。

## 5. 監控 + 告警

| 指標 | 正常值 | 告警 |
|---|---|---|
| RAID 剩餘空間 | > 500GB | < 200GB → 通知 |
| 最後一次 RAID 同步 | < 26 小時前 | > 26 小時前 → 通知 |
| 最後一次 OneDrive 同步 | < 8 天前 | > 8 天前 → 通知 |
| Qdrant snapshot 失敗率 | 0% | > 5% (7 天滑動) → 通知 |
| Mnemosyne DB 大小 | < 5GB | > 10GB → 通知(可能有 bloat) |

告警方式: 走 macOS `osascript` 顯示原生通知,或寄信到自架 SMTP。

## 6. 還原程序

詳細演練見 [`還原演練.md`](還原演練.md)。簡述:

| 情境 | 還原路徑 | 預估時間 |
|---|---|---|
| 單一檔案遺失 | L1 → L0 拉回 | < 1 分鐘 |
| 整個 L0 + L1 掛了 | L2 (RAID) → 工作站 | < 30 分鐘 |
| 整個 L2 掛了 | L3 (OneDrive) → 工作站 | 1-3 小時(取決於網速) |
| Mnemosyne 損壞 | L2 備份 → 還原 | < 5 分鐘 |
| Qdrant 損壞 | L2 snapshot → qdrant restore | 10-30 分鐘 |
| 整個工作站掛了 | 買新機器 + L3 + L4 (git) | 4-8 小時 |

## 7. v4 計畫衍生的特殊考量

### 7.1 加密與隱私

- L3 一定要用 rclone crypt(OneDrive 雖然本身有加密,但額外加一層確保微軟員工看不到檔名)
- 加密金鑰存 1Password,**不要**存在第二腦的 archive 裡(避免「加密的東西被解密後找不回去」的遞迴問題)
- 如果不想用雲端,可改用外接硬碟 + 親友家(俗稱「親友雲」);L3 邏輯不變

### 7.2 雙機一致性

- M1 Air 上的 `inbox/` 透過 Tailscale + `rsync -e ssh` 拉到工作站 L0
- 不直接從 M1 同步到 L2——L2 統一由工作站管,避免「兩台機器同時對 RAID 寫入」的 race condition
- M1 端的 SQLite 與 Qdrant 是「唯讀鏡像」(見 `systemd/ai-brain-qdrant-mirror.service`)

### 7.3 容量規劃

| 層級 | 預估 1 年後 | 預估 5 年後 |
|---|---:|---:|
| L0 + L1 | 200GB | 800GB |
| L2 (RAID) | 250GB (含備份) | 1.2TB |
| L3 (OneDrive) | 200GB (rolling 90 天) | 400GB |
| L4 (git) | < 100MB (純程式碼) | < 200MB |

> 4TB RAID 在 5 年後仍會有 2.5TB+ 剩餘空間,無虞。

---

## 8. 不要做的事 (Anti-patterns)

- ❌ 用 `rsync --delete` 跨層級(會把 L1 的刪除同步到 L2)
- ❌ 把 Mnemosyne 的 DB 直接放在雲端(裡面有對話紀錄,即使加密也太敏感)
- ❌ 把 InboxWatcher 的建議當指令(永遠 confirm 後再搬)
- ❌ 為了省空間把 L2 與 L3 合併(失去「異地」意義)
- ❌ 把 1Password 主密碼存在第二腦裡(邏輯遞迴爆炸)
