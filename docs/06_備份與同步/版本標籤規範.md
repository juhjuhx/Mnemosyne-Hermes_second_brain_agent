# 版本標籤規範 (Version Tag Conventions)

> **目標**: 在 `git tag`、`_archive/` 資料夾、Mnemosyne 內部 metadata 三處用一致的命名規則,避免「這個版本是什麼時候、對應哪個 phase」這類問題。

---

## 0. 摘要 (TL;DR)

```
v{MAJOR}.{MINOR}.{PATCH}-{YYYY-MM-DD}-{PHASE_TAG}[-{VARIANT}]
```

範例:
- `v4.0.0-2026-06-05-phase1`        — Phase 1 完成
- `v4.0.0-2026-06-15-phase2-deployed` — Phase 2 部署後
- `v4.1.0-2026-07-01`               — bugfix release
- `v5.0.0-2026-12-31-year-end`      — 年末歸檔

---

## 1. 三個軸: SemVer × Date × Phase

### 1.1 SemVer (v{MAJOR}.{MINOR}.{PATCH})

- **MAJOR**: 架構變更(例如 v3 → v4 是「從自建 agent 改用 Hermes Agent」)
- **MINOR**: 新增 phase、新增技能、新增資料源
- **PATCH**: bug fix、文件更新、相容性改進

範例:
- `v4.0.0`  初版 v4 計畫
- `v4.1.0`  新增 Phase 2 影片索引管線
- `v4.1.1`  修正 `qdrant-server` 對空 query 的處理
- `v5.0.0`  v5 重大改版(若未來發生)

### 1.2 日期 (YYYY-MM-DD)

- ISO 8601 格式
- **意義**: 該版本的「凍結日」
- 與 git commit date 應該一致(若有差異,以 commit date 為準)

### 1.3 Phase 標籤 (PHASE_TAG)

| 標籤 | 意義 |
|---|---|
| `phase0` | Phase 0 完成(啟動手冊) |
| `phase1` | Phase 1 完成(部署) |
| `phase2` | Phase 2 完成(索引流水線) |
| `phase3` | Phase 3 完成(Hermes 整合) |
| `phase4` | Phase 4 完成(評估) |
| `phase5` | Phase 5 完成(進階) |
| `hotfix-{DATE}` | 非計劃中的緊急修復 |
| `pre-{NEXT_VERSION}` | 下一個主要版本的 pre-release |
| `year-end-{YYYY}` | 年末歸檔(每年 12/31) |
| `experiment-{SHORT_DESC}` | 實驗性質,可能不會合併 |

### 1.4 VARIANT (可選)

修飾詞,只在少數情況下使用:
- `deployed`  — 已部署到 production
- `eval`      — 評估階段用的 tag(尚未穩定)
- `wip`       — work in progress
- `archived`  — 已歸檔,不再維護

## 2. git tag 與 _archive 對應表

| 事件 | git tag | _archive 資料夾 |
|---|---|---|
| Phase 0 完成 | `v4.0.0-2026-06-05-phase0` | `_archive/2026-06-05-phase0/` |
| Phase 1 完成 | `v4.0.0-2026-06-12-phase1` | `_archive/2026-06-12-phase1/` |
| 緊急修復 | `v4.0.1-2026-06-13-hotfix-qdrant-empty-query` | `_archive/2026-06-13-hotfix-qdrant/` |
| 半年回顧 | `v4.0.0-2026-12-31-mid-year` | `_archive/2026-12-31-mid-year/` |

> **規則**: `git tag` 與 `_archive/` 必須 *同步建立*——打 tag 的當下,`_archive/` 也寫入對應資料夾。

## 3. Mnemosyne 內部 metadata 標記

在 Mnemosyne 寫入對話紀錄時,加上 `plan_version` 與 `phase` 兩個 metadata 欄位:

```python
mnemosyne.add_memory(
    text="User: How do I configure Tailscale? Assistant: ...",
    metadata={
        "plan_version": "v4.0.0",
        "phase": "phase1",
        "tag_date": "2026-06-12",
        "machine": "m1",
    }
)
```

> 這讓 v5 升級時,可以查詢「在 v4.0.0 phase1 時,使用者問過哪些問題」,作為「v4 行為是否被顛覆」的回歸基準。

## 5. 不允許的命名

- ❌ `v4-final`        — 永遠不會是 final
- ❌ `v4-stable`       — 沒有任何版本是「穩定」的,只有「當下能跑」
- ❌ `latest`          — 與 git default 衝突
- ❌ `v4-2026`         — 缺少月日,精度不夠
- ❌ `v4.0.0-rc1`      — 不要用 SemVer 慣例的 rc 編號;改用 `v4.0.0-2026-06-12-pre-v4.1`

## 6. 自動產生 tag 的腳本

放在 `src/scripts/tag_release.sh`:

```bash
#!/usr/bin/env bash
# tag_release.sh — Create a git tag + _archive folder with a single command
#
# Usage:
#   ./tag_release.sh 4.0.0 phase1 2026-06-12
#   ./tag_release.sh 4.1.0 "" 2026-07-01       # no phase tag

set -euo pipefail
VERSION="$1"
PHASE_TAG="$2"
DATE="$3"
VARIANT="${4:-}"

[[ -z "$VERSION" || -z "$DATE" ]] && { echo "usage: $0 VERSION PHASE_TAG DATE [VARIANT]"; exit 1; }

SUFFIX="$DATE"
[[ -n "$PHASE_TAG" ]] && SUFFIX="${SUFFIX}-${PHASE_TAG}"
[[ -n "$VARIANT" ]]  && SUFFIX="${SUFFIX}-${VARIANT}"

TAG="v${VERSION}-${SUFFIX}"
ARCHIVE="_archive/${DATE}${PHASE_TAG:+-${PHASE_TAG}}${VARIANT:+-${VARIANT}}"

echo "→ git tag: $TAG"
echo "→ archive: $ARCHIVE"

git tag -a "$TAG" -m "Release $TAG (see CHANGELOG.md)"
mkdir -p "$ARCHIVE"
cp -r docs/ src/ tests/ README.md LICENSE LICENSE-DOCS CHANGELOG.md "$ARCHIVE/"
git add _archive/ CHANGELOG.md
git commit -m "Release $TAG"
git push origin "$TAG"

echo "✓ Tagged: $TAG"
```
